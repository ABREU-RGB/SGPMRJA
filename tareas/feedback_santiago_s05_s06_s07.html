¿Qué causa los conflictos?

Cuando dos ramas tocan el mismo archivo en las mismas líneas, git no puede decidir solo qué versión usar. Esto ocurre cuando:

Punto de partida: enmanuel en el momento que creaste S-07 enmanuel ──●────────────────────────────────────────── │ ├── S-06 (modifica ClienteController + clientes/index) │ └── S-07 (modifica los MISMOS archivos) ────────────────────────────────────────────────────────── Conflicto: git no sabe cuál de las dos versiones usar El TL resuelve manualmente, combinando ambas
📌 Importante
Que haya conflictos no significa que hayas hecho algo mal. Es una consecuencia natural de dos ramas que tocan el mismo módulo. El Tech Lead resuelve los conflictos — tú no tienes que hacer nada adicional con estas ramas.

💡 Recomendación
Antes de marcar una tarea como "terminada", hacerse estas preguntas:
  • ¿Esta funcionalidad aplica a otros módulos además del que toqué? Si sí, ¿los cubrí todos?
  • ¿El bug que arreglé aparece en otros lugares del mismo archivo? ¿Los revisé?
  • ¿Al eliminar algo del HTML quedó algún listener o función JS que ya no tiene referencia?

Reglas para las Próximas Tareas

📋 Recuerda siempre

1. Verificar cobertura completa antes de marcar como terminado
Si la tarea dice "en todos los módulos", buscar con grep todos los archivos afectados. No asumir que con cubrir uno o dos ya está.
2. Buscar ocurrencias del mismo patrón en todo el archivo
Si corriges un bug, usar Ctrl+F o grep para encontrar otras instancias del mismo problema. No hacer fix de una sola línea si hay 10 iguales.
3. Al eliminar HTML, limpiar el JS asociado
Si quitás un botón o modal, buscar sus event listeners (data-id, $('.remove-item-btn'), etc.) y eliminarlos también.
4. Mantener ramas actualizadas con enmanuel
Antes de empezar una tarea larga, hacer git pull origin enmanuel desde la rama de feature para reducir conflictos futuros.
