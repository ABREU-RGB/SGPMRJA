Deficiencia #1 — Sin boton "Restaurar"  Critica

Descripcion del problema

S-06 implemento la inhabilitacion logica (SoftDelete) con un boton que cambia "Eliminar" por "Inhabilitar" y una vista de historial para ver los registros inhabilitados. Sin embargo, no existe forma de restaurar un registro inhabilitado desde la interfaz. La inhabilitacion es un camino de ida.

Impacto directo
Si un usuario inhabilita un cliente, proveedor o producto por error, no tiene forma de revertirlo desde el sistema. La unica solucion seria intervenir directamente en la base de datos (UPDATE ... SET deleted_at = NULL), lo cual no es aceptable en produccion.

Estado actual vs. estado esperado

ESTADO ACTUAL (S-06): Registro activo ──[Inhabilitar]──> Registro en historial ──> ??? (sin salida) (solo boton "Ver") ESTADO ESPERADO: Registro activo ──[Inhabilitar]──> Registro en historial ──[Restaurar]──> Registro activo (boton "Ver" + boton "Restaurar")

Que se necesita implementar

Backend — Agregar metodo restore() en cada controller:

// En ClienteController, ProductoController, ProveedorController public function restore($id) { $cliente = Cliente::onlyTrashed()->findOrFail($id); $cliente->restore(); return response()->json([ 'message' => 'Cliente restaurado exitosamente.' ]); }

Rutas — Agregar ruta en web.php:

// Una ruta por cada modulo Route::post('clientes/{id}/restore', [ClienteController::class, 'restore']) ->name('clientes.restore'); Route::post('productos/{id}/restore', [ProductoController::class, 'restore']) ->name('productos.restore'); Route::post('proveedores/{id}/restore', [ProveedorController::class, 'restore']) ->name('proveedores.restore');

Frontend — Agregar boton "Restaurar" en generateButtons() cuando isTrashed === true:

// Dentro del bloque if (isTrashed) de cada vista if (isTrashed) { return '<div class="d-flex gap-1 justify-content-center">' + '<button class="btn btn-sm btn-soft-secondary view-item-btn" ...>' + ' <i class="ri-eye-fill"></i>' + '</button>' + '<button class="btn btn-sm btn-soft-success restore-item-btn" data-id="' + id + '">' + ' <i class="ri-arrow-go-back-line"></i>' + '</button>' + '</div>'; }

Deficiencia #2 — Catalogos maestros sin UI de inhabilitacion  Media

Descripcion del problema

La migracion BA-02 (ejecutada el 20/03/2026) agrego SoftDeletes (deleted_at) a 6 tablas de catalogos maestros. S-06 implemento la UI de inhabilitacion/historial para Clientes, Proveedores y Productos, pero no toco los catalogos maestros.

ModeloTabla DBSoftDelete en DBUI InhabilitarUI Historial
Bancobanco Si (BA-02) No No
Colorcolor Si (BA-02) No No
Tallatalla Si (BA-02) No No
TipoProductotipo_producto Si (BA-02) No No
Logologo Si (BA-02) No No
BordadoUbicacionbordado_ubicacion Si (BA-02) No No
Impacto
Actualmente, al "eliminar" un catalogo maestro desde la UI, el registro recibe un deleted_at gracias al trait SoftDeletes del modelo, pero el usuario no recibe feedback visual de que fue una inhabilitacion y no una eliminacion permanente. Tampoco hay vista de historial para estos catalogos.
Contexto
Estos son catalogos de configuracion (listas desplegables, opciones de formulario). Su flujo operacional es mucho mas simple que Clientes o Proveedores. El Tech Lead evaluara si justifica una UI de historial completa o si basta con:
  • Cambiar el texto del SweetAlert de "Eliminar" a "Inhabilitar" (minimo)
  • Agregar vista de historial solo donde tenga sentido operacional

Observacion #4 — setTimeout fragil en S-05  Baja

Al editar un proveedor juridico, el formulario carga el estado territorial y luego dispara un change para poblar los municipios. El municipio se selecciona con un setTimeout(100ms):

// resources/views/admin/proveedores/index.blade.php if (data.estado_territorial) { $("#estado-territorial-jur-field").val(data.estado_territorial); $("#estado-territorial-jur-field").trigger('change'); setTimeout(function () { $("#ciudad-jur-field").val(data.ciudad); // <-- depende de que el change ya haya poblado el select }, 100); // <-- 100ms arbitrarios }
Riesgo
Si el navegador esta bajo carga o el array de municipios es grande, 100ms podria no ser suficiente y el municipio no se pre-seleccionaria. La solucion ideal seria poblar el select de municipios de forma sincronica (ya que getMunicipios() es una funcion local, no una llamada AJAX) y luego seleccionar el valor. No bloquea el uso actual, pero es deuda tecnica.

Solucion recomendada

// Reemplazar el setTimeout por ejecucion sincronica: if (data.estado_territorial) { $("#estado-territorial-jur-field").val(data.estado_territorial); // Poblar municipios directamente (sincrono) var municipios = getMunicipios(data.estado_territorial); var select = $("#ciudad-jur-field"); select.empty().append('<option value="">Seleccione municipio</option>'); municipios.forEach(function(m) { select.append('<option value="' + m + '">' + m + '</option>'); }); select.val(data.ciudad); // Ya esta poblado, seleccion inmediata }

Observacion #5 — Toggle historial redundante en Clientes  Baja

Tras el merge de S-06 y S-07, el modulo de Clientes tiene dos formas de ver registros inactivos:

MecanismoOrigenCómo funciona
Boton "Ver Historial (Inactivos)" S-06 Navega a ?historial=true, que pre-selecciona el filtro estatus en "Inactivo"
Filtro desplegable "Estatus → Inactivo" S-07 Filtra la tabla en la misma pagina sin recargar
Impacto
No causa ningun bug — ambos mecanismos funcionan correctamente. Sin embargo, es una redundancia de UX. Al resolver el merge, el TL conecto el toggle con el filtro (cuando llega ?historial=true, pre-selecciona "Inactivo" en el dropdown).

Recomendacion: En una futura limpieza de UI, evaluar si el toggle de historial en Clientes se puede eliminar (ya que el filtro lo cubre), manteniendo el toggle solo en Proveedores y Productos donde no hay filtros avanzados.
