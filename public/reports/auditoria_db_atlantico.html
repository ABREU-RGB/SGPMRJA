Fortalezas del Esquema

✅ Arquitectura Persona como entidad base para User, Cliente, Empleado — diseño de herencia de tabla concreta correcto y escalable

Hallazgos Críticos

Riesgos de corrupción de datos, violaciones de integridad referencial o anti-patrones graves de arquitectura

6 hallazgos
CR-01 color y talla almacenados como texto libre — tablas lookup ignoradas
📋 Tablas afectadas: detalle_cotizacion · detalle_pedido
-- detalle_cotizacion `color` varchar(50) DEFAULT NULL, -- texto libre, sin FK `talla` varchar(191) DEFAULT NULL, -- ¡varchar(191)! -- detalle_pedido (mismo problema, tamaño diferente) `color` varchar(50) DEFAULT NULL, `talla` varchar(50) DEFAULT NULL, -- varchar(50) aquí

El sistema posee las tablas colores (32 registros) y tallas (15 registros) completamente pobladas con datos normalizados. Sin embargo, los detalles de cotización y pedido almacenan estos valores como cadenas de texto libre sin foreign key. Esto permite que un pedido tenga talla = "mediano" en lugar de "M", haciendo que los reportes de producción por talla o color sean estadísticamente poco confiables.

Además, hay una inconsistencia de tipos entre las dos tablas "gemelas": detalle_cotizacion.talla es varchar(191) mientras detalle_pedido.talla es varchar(50).

Tipo de riesgo: Integridad de Datos 2NF Violation
CR-02 proveedor semi-normalizado: proveedores jurídicos fuera del sistema Persona
📋 Tabla afectada: proveedor
`persona_id` bigint DEFAULT NULL, -- solo para tipo 'natural' `razon_social` varchar(100) DEFAULT NULL, -- texto libre (juridico) `direccion` varchar(191) DEFAULT NULL, -- fuera del sistema direccion `telefono` varchar(20) DEFAULT NULL, -- fuera del sistema telefono `email` varchar(191) DEFAULT NULL, -- duplicado `contacto` varchar(100) DEFAULT NULL, `telefono_contacto` varchar(20) DEFAULT NULL,

cliente fue completamente normalizado a persona + telefono + direccion. proveedor quedó a la mitad: los proveedores naturales usan persona_id, pero los jurídicos (8 de 14 en producción) tienen sus datos de contacto embebidos directamente — dirección como texto libre, teléfono y email duplicados fuera del sistema normalizado. No es posible buscar un proveedor desde persona ni listar sus teléfonos desde la tabla telefono.

Tipo de riesgo: Inconsistencia Arquitectural 2NF / 3NF
CR-03 pedido.estado y cotizacion.estado son varchar libre — sin restricción ENUM
📋 Tablas afectadas: pedido · cotizacion (campos estado y prioridad)
-- pedido y cotizacion — VARCHAR sin restricción `estado` varchar(191) NOT NULL DEFAULT 'Pendiente', `prioridad` varchar(191) NOT NULL DEFAULT 'Normal', -- orden_produccion — ENUM correcto `estado` enum('Pendiente','En Proceso','Finalizado','Cancelado')

La tabla orden_produccion usa correctamente ENUM para su estado. Las tablas pedido y cotizacion usan varchar(191). Estados observados en los datos: 'Pendiente', 'Procesando', 'Completado', 'Cancelado' para pedido. Un typo como 'Completada' o 'completado' se inserta silenciosamente sin error de base de datos, rompiendo filtros y reportes de estado.

Tipo de riesgo: Datos Corruptos Silenciosos
CR-04 user.persona_id nullable — usuarios activos sin entidad Persona
📋 Tabla afectada: user
`persona_id` bigint DEFAULT NULL, -- users 5 y 7 son NULL (admins activos) `name` varchar(191) NOT NULL, -- duplica persona.nombre + apellido `email` varchar(191) NOT NULL, -- duplica persona.email

Hay 2 usuarios administradores activos reales (id=5 Vanessa Díaz, id=7 Jesús Rodríguez) con persona_id = NULL. Esto viola el contrato de la arquitectura normalizada: si toda la información de la persona vive en la tabla persona, user no debería tener campos name y email propios. Existen dos fuentes de verdad para el nombre e email de un usuario. Si se actualiza el email en persona pero no en user, el sistema de autenticación quedará desincronizado.

Tipo de riesgo: Datos Duplicados / Desincronización Anti-patrón Arquitectural
CR-05 orden_produccion.logo: texto libre desconectado de la tabla logos
📋 Tabla afectada: orden_produccion
`logo` varchar(191) DEFAULT NULL, -- Datos reales: 'Los Caminos', 'Asoportuguesa Corp', 'Logo UPTP' -- 'Logo UPTP' NO existe en la tabla logos → integridad rota

La tabla logos existe con IDs, nombres únicos y filenames normalizados (20 logos). Sin embargo, orden_produccion.logo almacena el nombre del logo como texto libre sin FK. En los datos de producción existe el valor 'Logo UPTP' que no corresponde a ningún registro en la tabla logos — integridad referencial rota a nivel de negocio. Esto también imposibilita obtener el filename del logo para imprimir órdenes de producción.

Tipo de riesgo: Integridad Referencial Rota
CR-06 movimiento_insumo.created_by: FK sin comportamiento ON DELETE explícito
📋 Tabla afectada: movimiento_insumo
`created_by` bigint unsigned NOT NULL, -- Sin ON DELETE → InnoDB asume RESTRICT implícito CONSTRAINT movimientos_insumos_created_by_foreign FOREIGN KEY (created_by) REFERENCES user (id) -- ← falta ON DELETE RESTRICT explícito

created_by es NOT NULL. Si se intenta eliminar un usuario que haya registrado movimientos de inventario, la operación fallará con un error de FK en producción — probablemente sin un mensaje claro para el usuario final. El comportamiento debe ser ON DELETE RESTRICT explícito (los movimientos de inventario son registros contables inmutables), o cambiar created_by a nullable con ON DELETE SET NULL para preservar el historial auditando al "usuario eliminado".

Tipo de riesgo: Error en Producción

ME-01 8 índices faltantes en columnas de alta frecuencia de consulta

En un ERP de manufactura, los siguientes filtros y ordenamientos ocurrirán en cada carga de pantalla. Sin índice, MySQL realizará full table scan:

Tabla Columna faltante Query típica
pedidoestadoWHERE estado = 'Pendiente'
pedidofecha_pedidoORDER BY fecha_pedido DESC
cotizacionestadoWHERE estado != 'Convertida'
orden_produccionestadoWHERE estado IN ('Pendiente','En Proceso')
orden_produccionfecha_fin_estimadaAlertas de órdenes vencidas
movimiento_insumocreated_atORDER BY created_at DESC por insumo
tasa_cambiofecha_bcvORDER BY fecha_bcv DESC LIMIT 1
insumostock_actualWHERE stock_actual <= stock_minimo
Tipo de riesgo: Rendimiento
ME-02 4 tablas recientes violan la convención singular establecida por el propio proyecto

La migración 2025_12_04_150028_rename_all_tables_to_singular_final renombró todas las tablas a singular. Las tablas creadas después retornaron al plural sin consistencia:

-- Tablas en PLURAL (incorrecto para este proyecto) colores → debería ser: color tallas → debería ser: talla logos → debería ser: logo bordado_ubicaciones → debería ser: bordado_ubicacion

Los modelos Eloquent Color, Talla, Logo probablemente tienen protected $table hardcodeado para compensar, generando deuda técnica.

Tipo de riesgo: Convención Laravel
ME-03 tipo_producto.contador viola 3NF — dato derivado almacenado como columna
`contador` int NOT NULL DEFAULT '0' -- Valor real: Franela tiene contador=0 pero hay 2 productos tipo Franela -- Dato correcto: SELECT COUNT(*) FROM producto WHERE tipo_producto_id = 2

El campo contador almacena cuántos productos existen de ese tipo — un dato completamente derivable con COUNT(*). Los datos actuales ya muestran desincronización: Franela tiene contador = 0 pero existen 2 productos Franela (FRN-CEV-001, FRN-CUR-001). Cualquier operación directa en DB o seed rompe el contador.

Tipo de riesgo: 3NF Violation
ME-04 produccion_diaria sin campo fecha_produccion — registros retroactivos imposibles
-- No existe fecha_produccion en la tabla `created_at` timestamp NULL DEFAULT NULL -- se usa como fecha de producción -- Problema: supervisor registra hoy la producción de ayer -- created_at = NOW() pero la producción fue ayer

Un supervisor que registre producción del día anterior debe hacerlo con created_at = NOW() ya que no existe un campo de fecha de producción separado. Los reportes de eficiencia diaria quedan distorsionados con registros retroactivos. Se necesita un campo fecha_produccion DATE NOT NULL independiente del timestamp de creación del registro.

Tipo de riesgo: Reportes Poco Confiables
ME-05 pedido.banco_id es un campo legacy muerto con índice y FK activos
`banco_id` bigint DEFAULT NULL, -- LEGACY: NULL en 58/58 pedidos `banco_transferencia_id` bigint DEFAULT NULL, -- reemplazó a banco_id `banco_pago_movil_id` bigint DEFAULT NULL, -- reemplazó a banco_id

La migración add_separate_bank_fields_to_pedidos_table agregó los campos diferenciados por método de pago, pero banco_id nunca fue eliminado. En los 58 pedidos actuales banco_id = NULL en el 100% de los casos. El campo está muerto pero consume espacio en el índice y genera confusión al leer el esquema.

Tipo de riesgo: Deuda Técnica
ME-06 orden_produccion sin fecha_fin_real — métricas de eficiencia imposibles
`fecha_fin_estimada` date NOT NULL, -- existe -- fecha_fin_real date NULL ← NO EXISTE -- Cuando estado → 'Finalizado' solo se actualiza updated_at

Sin fecha_fin_real no es posible calcular: días de retraso por orden, cumplimiento de fechas estimadas, velocidad real de producción por empleado/producto, ni tendencias de eficiencia operativa. El campo updated_at actualmente se usa como proxy, pero puede cambiar por cualquier edición posterior al cierre de la orden.

Tipo de riesgo: KPIs de Producción Faltantes
ME-07 detalle_pedido_insumo: propósito ambiguo duplica la lógica de detalle_orden_insumo
-- Tabla 1: insumos por ítem de pedido (antes de producción) detalle_pedido_insumo: detalle_pedido_id → insumo_id + cantidad_estimada -- Tabla 2: insumos por orden de producción (durante producción) detalle_orden_insumo: orden_produccion_id → insumo_id + cantidad_estimada + cantidad_utilizada

El flujo de negocio correcto es Pedido → Orden de Producción → consume Insumos. La relación directa detalle_pedido_insumo crea un cortocircuito que salta la orden, generando ambigüedad sobre cuál tabla es la fuente de verdad para el consumo de materiales. Posible residuo de arquitectura pre-refactoring.

Hallazgos de Severidad Baja

Ajustes de nomenclatura, tipos de datos subóptimos o inconsistencias menores

6 hallazgos
BA-01 Nombre de índice legacy en produccion_diaria referencia "operario" en lugar de "empleado"
KEY `produccion_diaria_operario_id_foreign` (`empleado_id`) -- El índice apunta a empleado_id pero su nombre dice "operario" -- Residuo de la migración 2026_03_06_155710_rename_operario_id_to_empleado_id

Funcional, pero confuso para cualquier DBA o desarrollador que inspeccione el esquema. El índice fue renombrado como columna pero no como nombre de KEY.

Tipo de riesgo:Nomenclatura
BA-02 insumo.tipo ENUM contiene 'Botón' con tilde — riesgo en imports/exports
`tipo` enum('Tela','Hilo','Botón','Cierre','Etiqueta','Otro') -- 'Botón' tiene tilde → problemático en scripts externos sin charset -- Recomendado: 'Boton' sin acento diacrítico en valores ENUM

Los valores de ENUM con caracteres especiales pueden causar problemas en importaciones CSV, scripts de migración externos o integraciones API que no especifiquen correctamente el charset.

Tipo de riesgo:Compatibilidad
BA-03 persona.estado_persona nombrado confusamente — almacena estado geográfico
`estado_persona` varchar(100) DEFAULT NULL -- Datos reales: 'Portuguesa', 'Washington DC', 'Texas' -- El campo almacena estado geográfico, no estado de la persona (activo/inactivo) -- Recomendado: estado_geografico o estado_ubicacion

El nombre estado_persona es semánticamente ambiguo: sugiere "estado" como activo/inactivo de la persona, pero almacena el estado o territorio geográfico de residencia.

Tipo de riesgo:Nomenclatura
BA-04 user.avatar definido como TEXT — varchar(255) es suficiente y más eficiente
`avatar` text COLLATE utf8mb4_unicode_ci, -- Datos reales: 'avatars/69a05e0f7203f.png' (27 caracteres) -- text = hasta 65,535 bytes. varchar(255) es más que suficiente. -- text no puede ser indexado directamente en MySQL

Las rutas de archivo de avatar no superan los 100 caracteres. Usar text impide crear índices directos sobre este campo si fuera necesario.

Tipo de riesgo:Tipo Subóptimo
BA-05 password_resets sin PRIMARY KEY — tabla legada de Laravel anterior
CREATE TABLE `password_resets` ( `email` varchar(191) NOT NULL, `token` varchar(191) NOT NULL, `created_at` timestamp NULL, KEY password_resets_email_index (email) -- sin PK ) -- Laravel 9+ usa password_reset_tokens con PK compuesto

La tabla no tiene PRIMARY KEY. Funcional en volúmenes bajos, pero degradará en entornos de alta concurrencia de resets de contraseña.

Tipo de riesgo:Rendimiento Marginal
BA-06 banco.nombre = varchar(191) sobredimensionado — varchar(100) es suficiente
`nombre` varchar(191) NOT NULL -- El nombre de banco más largo en Venezuela: ~60 caracteres -- varchar(191) es el default ciego de Laravel. varchar(100) es correcto.

Menor impacto, pero refleja el hábito de usar varchar(191) como tamaño por defecto sin analizar el dominio real del dato.

🎯 Plan de Acción Recomendado — Orden de Prioridad

1
CR-03 — ENUM en pedido.estado y cotizacion.estado — Cambio de bajo riesgo, alta recompensa. Una migración de 30 minutos elimina el riesgo de estados inválidos silenciosos que afectan el corazón del negocio.
2
CR-01 — color_id y talla_id como FK en detalles — Las tablas lookup ya existen y están pobladas. Solo falta agregar las FK y migrar los datos. Impacto directo en confiabilidad de reportes de producción.
3
ME-01 — Crear los 8 índices faltantes — Una migración de índices es no destructiva, puede correr en producción sin downtime, y el impacto en rendimiento es inmediato.
4
ME-04 + ME-06 — Agregar fecha_produccion y fecha_fin_real — Columnas nullable, migración segura. Desbloquea los reportes de KPI de producción que son el núcleo del módulo de manufactura.
5
CR-02 — Normalizar proveedor jurídico — Mayor esfuerzo pero necesario para coherencia arquitectural. Requiere migración de datos de los 8 proveedores jurídicos al sistema persona/telefono/direccion.
