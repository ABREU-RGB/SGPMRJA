Sección 03 Hallazgos Críticos

🔴 CRÍTICO — Riesgo de pérdida de datos o integridad rota 7 hallazgos

CR-01 ON DELETE CASCADE en cotizacion.user_id y pedido.user_id

Tablas: cotizacion, pedido Tipo: Foreign Key destructiva
Impacto

Borrar un usuario del sistema elimina permanentemente todas sus cotizaciones y pedidos, incluyendo detalles, bordados asociados (CASCADE en cadena), y las órdenes de producción quedan huérfanas con pedido_id = NULL. Es la FK más peligrosa del schema — destruye datos financieros y comerciales irrecuperables.

Agravante: La tabla user NO tiene SoftDeletes ni columna deleted_at. No hay red de seguridad.

Implementación Propuesta 
// Migración — cambiar CASCADE a RESTRICT
Schema::table('cotizacion', function (Blueprint $table) {
    $table->dropForeign('cotizaciones_user_id_foreign');
    $table->foreign('user_id')->references('id')->on('user')
          ->onDelete('restrict');
});

Schema::table('pedido', function (Blueprint $table) {
    $table->dropForeign('pedidos_user_id_foreign');
    $table->foreign('user_id')->references('id')->on('user')
          ->onDelete('restrict');
});

CR-02 ON DELETE CASCADE en detalles de producto — destruye historial comercial

Tablas: detalle_cotizacion, detalle_pedido Tipo: Foreign Key destructiva
Impacto

Si se hace hard-delete de un producto, se eliminan todas las líneas de detalle de cotizaciones y pedidos donde aparecía. El total del pedido/cotización padre queda inconsistente (suma de detalles ≠ total). Producto usa SoftDeletes, pero si alguien ejecuta forceDelete() o un DELETE directo, se pierde el historial.

Implementación Propuesta 
// Cambiar ambas a ON DELETE RESTRICT
Schema::table('detalle_cotizacion', function (Blueprint $table) {
    $table->dropForeign('detalle_cotizaciones_producto_id_foreign');
    $table->foreign('producto_id')->references('id')->on('producto')
          ->onDelete('restrict');
});

Schema::table('detalle_pedido', function (Blueprint $table) {
    $table->dropForeign('detalle_pedidos_producto_id_foreign');
    $table->foreign('producto_id')->references('id')->on('producto')
          ->onDelete('restrict');
});

CR-03 ON DELETE CASCADE en movimiento_insumo.insumo_id — pérdida de trazabilidad

Tabla: movimiento_insumo Tipo: Foreign Key destructiva sobre registros de auditoría
Impacto

Borrar un insumo elimina todo el historial de movimientos de inventario. Los movimientos son registros de auditoría que documentan quién movió qué, cuándo y por qué. Deben ser inmutables.

Implementación Propuesta 
Schema::table('movimiento_insumo', function (Blueprint $table) {
    $table->dropForeign('movimientos_insumos_insumo_id_foreign');
    $table->foreign('insumo_id')->references('id')->on('insumo')
          ->onDelete('restrict');
});

CR-04 Relación incorrecta User::produccionDiaria() — FK apunta a tabla equivocada

Modelo: User.php Tabla: produccion_diaria Tipo: Desconexión Schema ↔ Eloquent
Impacto

La relación User::produccionDiaria() define hasMany(ProduccionDiaria::class, 'empleado_id'), pero el FK produccion_diaria.empleado_id referencia empleado(id), NO user(id). Si un User tiene id=3 y un Empleado diferente tiene id=3, retorna datos del empleado equivocado. Si no coinciden IDs, retorna resultado vacío silenciosamente.

Implementación Propuesta 
// En User.php — ELIMINAR la relación produccionDiaria()
// Acceso correcto: $user->persona->empleado->produccionDiaria

// Eliminar este método del modelo User:
public function produccionDiaria()
{
    return $this->hasMany(ProduccionDiaria::class, 'empleado_id');
}

CR-05 UNIQUE en persona.documento_identidad sin incluir tipo_documento

Tabla: persona Tipo: Constraint de unicidad incompleto
Impacto

Una persona con cédula V-12345678 y otra con pasaporte E-12345678 no pueden coexistir — el UNIQUE es solo sobre el número, no incluye el tipo de documento. También afecta proveedores jurídicos (J-) que podrían tener el mismo número que una persona natural.

Implementación Propuesta 
Schema::table('persona', function (Blueprint $table) {
    $table->dropUnique('persona_documento_identidad_unique');
    $table->unique(
        ['tipo_documento', 'documento_identidad'],
        'persona_tipo_doc_documento_unique'
    );
});

CR-06 pedido.banco_transferencia_id y banco_pago_movil_id sin FK constraint

Tabla: pedido Tipo: Integridad referencial rota
Impacto

Las columnas existen y el modelo Pedido define las relaciones bancoTransferencia() y bancoPagoMovil(), pero no hay CONSTRAINT FOREIGN KEY en el schema. Se puede guardar un banco_transferencia_id = 999 que no existe. Eloquent retorna null silenciosamente.

Implementación Propuesta 
Schema::table('pedido', function (Blueprint $table) {
    $table->foreign('banco_transferencia_id')
          ->references('id')->on('banco')
          ->onDelete('set null');

    $table->foreign('banco_pago_movil_id')
          ->references('id')->on('banco')
          ->onDelete('set null');
});

CR-07 Tabla user sin SoftDeletes — punto de falla sistémico

Tabla: user Modelo: User.php Tipo: Ausencia de protección
Impacto

La tabla user no tiene columna deleted_at y el modelo no usa SoftDeletes. Pero user es referenciada por cotizaciones (CASCADE), pedidos (CASCADE), órdenes (NO ACTION) y movimientos (RESTRICT). No hay manera segura de “desactivar” un usuario. El campo estado es un flag de aplicación, no una protección a nivel DB.

Tabla que referencia user FK ON DELETE Riesgo
cotizacion.user_id cotizaciones_user_id_foreign CASCADE Destruye cotizaciones
pedido.user_id pedidos_user_id_foreign CASCADE Destruye pedidos
orden_produccion.created_by ordenes_produccion_created_by_foreign NO ACTION Bloquea borrado
movimiento_insumo.created_by movimientos_insumos_created_by_foreign RESTRICT Bloquea borrado
Implementación Propuesta 
// Migración
Schema::table('user', function (Blueprint $table) {
    $table->softDeletes();
});

// Modelo User.php
use Illuminate\Database\Eloquent\SoftDeletes;

class User extends Authenticatable
{
    use SoftDeletes;
    // ...
}

Sección 04 Hallazgos de Severidad Media

🟡 MEDIO — Inconsistencias y riesgos operacionales 7 hallazgos

ME-01 ON DELETE CASCADE inconsistente en FKs de persona_id

Tablas: user, empleado, cliente, proveedor Tipo: Inconsistencia de política FK

Las cuatro tablas que referencian persona(id) usan políticas ON DELETE diferentes e inconsistentes:

Tabla FK → persona ON DELETE ¿Correcto?
user persona_id CASCADE Peligroso — borra usuario
empleado persona_id CASCADE Peligroso — borra empleado
cliente persona_id RESTRICT Correcto
proveedor persona_id SET NULL Peligroso — ver ME-02
telefono persona_id CASCADE OK (dato dependiente)
direccion persona_id CASCADE OK (dato dependiente)

Propuesta: Unificar user, empleado y proveedor a ON DELETE RESTRICT. Teléfonos y direcciones son dependientes de persona — CASCADE es correcto ahí.

ME-02 proveedor.persona_id ON DELETE SET NULL — proveedor queda vacío

Tabla: proveedor Tipo: FK con comportamiento dañino

El modelo Proveedor obtiene todo su contenido de persona vía accessors (nombre, documento, teléfono, email, dirección). Si persona_id se pone NULL, todos los accessors retornan null. El proveedor existe pero es un registro fantasma sin datos visibles.

Propuesta: Cambiar a ON DELETE RESTRICT.

ME-03 ON DELETE CASCADE en detalle_orden_insumo.insumo_id

Tabla: detalle_orden_insumo Tipo: Foreign Key destructiva silenciosa

Borrar un insumo elimina silenciosamente sus asignaciones a órdenes de producción. La orden queda con datos de costos estimados inconsistentes — se pierde la relación insumo-orden sin aviso.

Propuesta: Cambiar a ON DELETE RESTRICT.

ME-04 Índices compuestos faltantes para queries de alto tráfico

Tablas: varias Tipo: Rendimiento
Query Típico Índice Faltante Justificación
Historial movimientos por insumo movimiento_insumo(insumo_id, created_at) Cubre el JOIN + ORDER BY
Producción diaria por orden + fecha produccion_diaria(orden_id, fecha_produccion) Reportes de producción
Pedidos por cliente + estado pedido(cliente_id, estado) Dashboard de pedidos
Cotizaciones por cliente + estado cotizacion(cliente_id, estado) Mismo patrón
Órdenes por estado + fecha fin orden_produccion(estado, fecha_fin_estimada) Índices separados existen pero no compuesto

ME-05 ordenes_produccion_created_by_foreign sin ON DELETE explícito

Tabla: orden_produccion Tipo: Inconsistencia declarativa

La FK funciona correctamente (default = RESTRICT en InnoDB), pero es implícito. Contraste con movimiento_insumo.created_by que sí declara ON DELETE RESTRICT explícitamente. Debe ser explícito para documentar la intención del DBA.

ME-06 Naming inconsistente — tablas singular vs plural

Schema completo Tipo: Convención de nombrado
Singular (convención dominante) Plural (inconsistente)
banco, cliente, cotizacion, pedido, persona, producto, empleado, insumo, proveedor, telefono, direccion colores, tallas, logos, bordado_ubicaciones, failed_jobs, personal_access_tokens, password_resets

El proyecto usa convención singular mayoritariamente. Las tablas plurales son catálogos más recientes o tablas de framework (Laravel default). No es urgente pero genera confusión al escribir migraciones.

ME-07 detalle_cotizacion y detalle_pedido con nombre_logo desnormalizado

Tablas: detalle_cotizacion, detalle_pedido Tipo: Desnormalización residual

Ambas tablas tienen nombre_logo varchar(191) como texto libre, mientras que orden_produccion ya fue migrado a logo_id FK. Las cotizaciones y pedidos siguen usando texto libre para el logo — dato desnormalizado sin referencia a la tabla logos.

Propuesta futura: Migrar a logo_id FK con ON DELETE SET NULL, como se hizo en orden_produccion.

Sección 05 Hallazgos de Severidad Baja

🟢 BAJO — Mejoras de calidad y limpieza 5 hallazgos

BA-01 password_resets — PK en email + INDEX en email redundante

Tabla: password_resets Tipo: Índice redundante

El PRIMARY KEY ya es un índice. El KEY password_resets_email_index adicional ocupa espacio innecesariamente y ralentiza inserts sin beneficio en lecturas.

BA-02 Catálogos maestros sin SoftDeletes

Tablas: banco, colores, tallas, logos, bordado_ubicaciones, tipo_producto Tipo: Protección de datos

Estos catálogos no tienen deleted_at. Usan campo activo (los que lo tienen) como soft-flag de aplicación. Si se hace DELETE directo, las FKs con ON DELETE SET NULL dejan huecos y las de ON DELETE RESTRICT bloquean. Aceptable para catálogos pequeños pero sin red de seguridad a nivel DB.

BA-03 produccion_diaria.empleado_id FK sin ON DELETE explícito

Tabla: produccion_diaria Tipo: Inconsistencia declarativa

Mismo patrón que ME-05. Funciona como RESTRICT (default InnoDB), pero debería ser explícito para documentar intención.

BA-04 Campos de pago en pedido como columnas flat

Tabla: pedido Tipo: Diseño extensible

7 columnas para 3 métodos de pago: efectivo_pagado, transferencia_pagado, pago_movil_pagado, referencia_transferencia, banco_transferencia_id, referencia_pago_movil, banco_pago_movil_id. Si se agrega un nuevo método de pago (Zelle, cripto, etc.), hay que agregar más columnas.

Propuesta futura: Normalizar en tabla pago_pedido(id, pedido_id, metodo_pago ENUM, referencia, banco_id, monto).

BA-05 cotizacion.estado ENUM con inconsistencia gramatical

Tabla: cotizacion Tipo: Consistencia de datos 
-- cotizacion (femenino: "cotización cancelada")
estado ENUM('Pendiente','Aprobada','Cancelado','Convertida','Vencida')
--                                 ^ debería ser 'Cancelada'

-- pedido (masculino: "pedido cancelado")
estado ENUM('Pendiente','Procesando','Completado','Cancelado')
--                                                  ^ correcto

“Cancelado” en cotización debería ser “Cancelada” (femenino). Inconsistencia gramatical menor.

Sección 06 Plan de Remediación Priorizado

Orden recomendado de implementación, priorizando la protección de datos comerciales y la eliminación de riesgos de pérdida irreversible.

Estrategia: Agrupar correcciones relacionadas en migraciones batch para minimizar el número de deploys y asegurar consistencia transaccional.
  1. CR-01 + CR-07 — Agregar SoftDeletes a user + cambiar CASCADE→RESTRICT en cotizacion.user_id y pedido.user_id. Elimina la amenaza #1 del schema.
  2. CR-06 — Agregar FKs faltantes de banco en pedido (banco_transferencia_id, banco_pago_movil_id). Cierra integridad referencial rota.
  3. CR-04 — Eliminar relación incorrecta User::produccionDiaria(). Fix de código, sin migración DB.
  4. CR-05 — Cambiar UNIQUE de persona.documento_identidad a compuesto (tipo_documento, documento_identidad).
  5. CR-02 + CR-03 + ME-03 — Batch de cambios CASCADE→RESTRICT en detalles de producto, movimientos de insumo, y detalles de orden-insumo.
  6. ME-01 + ME-02 — Unificar políticas ON DELETE de persona_id: cambiar user, empleado y proveedor a RESTRICT.
  7. ME-04 — Crear índices compuestos para queries de alto tráfico.
  8. ME-05 + BA-03 — Hacer explícito el ON DELETE RESTRICT en orden_produccion.created_by y produccion_diaria.empleado_id.
  9. ME-07 — Migrar nombre_logo a logo_id FK en detalle_cotizacion y detalle_pedido (requiere data migration).
  10. BA-01 → BA-05 — Batch de correcciones menores: índice redundante, ENUM gramatical, documentación de naming.
